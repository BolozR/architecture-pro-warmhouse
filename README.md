### 1. Описание функциональности монолитного приложения

**Управление отоплением:**

- Пользователи могут удалённо включать/выключать отопление в своих домах
- Система поддерживает управление удалёнными рубильниками отопления в подключённых домах

**Мониторинг температуры:**

- Пользователи могут просматривать текущую температуру в своих домах через веб-интерфейс
- Система поддерживает получение данных температуре с датчиков, установленных в домах

### 2. Анализ архитектуры монолитного приложения

### **4. Проблемы монолитного решения**

- добавление новых клиентов через вызов мастера

В синхронном взаимодействии проблем не вижу, 1 сервер без спец оптимизаций сможет держать держать десятки тысяч соединений, 
каждый клиент с частотой опроса (едвали требуется чаще минуты) передаёт конечный и разумно ограниченный массив чисел 
(по хорошему начальное значение при установке соединения и short дифф во время сессии) уперётся в производительность.

Обновление даунтайма не требует если есть канареечное развёртывание и load balancer.

Постгресс на nvme такую нагрузку выдержит запросто, хотя можно сказать что нагрузка интенсивна на запись и нужно хранилище на LSM.
В буфферизации значений в памяти на стороне приложения не вижу смысла, только если частоту опроса сделать очень большой,
но в этом тоже смысла нет.

Возможно есть смысл добавить стендбай ноду для постгресса, хотя в одном экземпляре при текущей нагрузке он будет очень лёгкий
и быстро поднимится при падении, данные с счётчиков актуальны в пределах +- дня.
